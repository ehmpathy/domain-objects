# feat: stamp domain-object identity + key metadata onto `.schema` for cross-service introspection

## handoff from: sdk-aws-lambda · feat-sdk-codegen

### context

`sdk-aws-lambda` builds cross-service sdk codegen. it introspects a service's
lambda contracts (zod `schema` → json-schema via `z.toJSONSchema`) and
reconstructs the upstream service's domain-objects in the *consumer* repo — e.g.
`Job` from svc-jobs becomes `SvcJobsJob` under svc-gateway, with its key metadata
intact.

for this to work, a domain-object's identity and key metadata must **survive
serialization to json-schema**. today `z.toJSONSchema(SomeDobj.schema)` emits a
plain, anonymous object shape — it carries none of the domain-object's identity
or keys. so the consumer sees a nameless shape it cannot name or de-dupe.

### the ask

add a `.schema` getter on `DomainObject` classes that returns the author-supplied
`static schema` (zod) **stamped** so these ride through `z.toJSONSchema()`:

1. **identity marker** — an "i am a domain object named X" tag (the class name,
   and/or `alias`). this is the name-carrier: it lets a consumer detect that a
   given json-schema *is* a domain object, learn its name, and de-dupe the same
   dobj referenced by many endpoints.
2. **key metadata** — `primary`, `unique`, `alias`, `nested`.

these already live as statics on the class (`DomainObject.d.ts:41,50,61,97`); the
ask is the getter that merges them into the emitted schema, carried via zod
`.meta()` (or the registry) so they appear as a custom keyword — e.g.
`x-domain-object: { name, primary, unique, alias, nested }` — in the json-schema
output, and round-trip through serialize → json-schema → parse.

### usage that should work

```ts
// an upstream service declares an endpoint:
genLambdaEndpoint({
  schema: { output: z.object({ surfboard: SeaturtleSurfboard.schema }) },
  // ...
});
// SeaturtleSurfboard.schema auto-stamps its identity + metadata, so
// z.toJSONSchema carries:
//   x-domain-object: { name: 'SeaturtleSurfboard', primary, unique, alias, nested }
```

### what consumers do NOT need

consumers reconstruct full `DomainEntity` / `DomainLiteral` classes **with** the
metadata (`primary` / `unique` / `alias` / `nested`) but **without** the zod
`schema`. the originator service runtime-typechecks; consumers do not re-validate.
so the getter's job is to *carry* identity + metadata across the wire — not to
make consumers reproduce the validation schema.

### why it matters

without this, consumer-side codegen can only emit anonymous types — it cannot
name `SvcJobsJob`, de-dupe it across endpoints, or carry `.primary` / `.unique`
for ref + identity use.

### dependency

`sdk-aws-lambda`'s `feat-sdk-codegen` is gated on this. happy to collaborate on
the exact pragma shape (keyword name, nested-ref support, alias-as-name rules).
