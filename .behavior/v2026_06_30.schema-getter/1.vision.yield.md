# 1.vision — `.contract` for cross-service introspection

> illustrate the world implied by `0.wish.md`: a `.contract` getter that stamps
> domain-object identity + key metadata onto a zod schema so it survives
> `z.toJSONSchema()`, following the `_dobj` serialization precedent.

---

## ubiqlang: `schema` vs `contract`

> **`schema` validates. `contract` identifies.**

| term | what it is | answers | who uses it |
|---|---|---|---|
| **`Job.schema`** | the raw zod validation schema (author-supplied) | "is this data valid?" | the **originator**, at runtime, to typecheck |
| **`Job.contract`** | the schema **stamped** with identity + keys (`x-domain-object`) | "what *is* this, and how do I rebuild it?" | the **consumer**, across the wire, to name + de-dupe + reconstruct |

`schema ⊂ contract` — the schema is one ingredient of the contract. the contract is
the schema that knows its own name and keys. the wish draws this exact line: consumers
need the **contract** (identity to rebuild), not the **schema** (they do not re-validate).

## the outcome world

### day-in-the-life

an upstream service (`svc-jobs`) declares its lambda contracts with domain-object schemas:

```ts
genLambdaEndpoint({
  schema: { output: z.object({ job: Job.contract }) },
});
```

a downstream service (`svc-gateway`) runs sdk codegen. it introspects the upstream
contract, converts each schema to json-schema, and finds a pragma riding inside:

```jsonc
{
  "type": "object",
  "properties": { "job": { /* ...job shape... */ } },
  "x-domain-object": {
    "name": "Job",
    "primary": ["uuid"],
    "unique": ["exid"],
    "alias": { "singular": "job", "plural": "jobs" },
    "nested": { "customer": "Customer" }
  }
}
```

from that pragma alone the consumer can:
- **name** the reconstructed type (`SvcJobsJob`) instead of an anonymous shape
- **de-dupe** the same dobj referenced by many endpoints (match on `name`)
- **carry keys** (`primary` / `unique`) so the consumer rebuilds a real `DomainEntity`
  with ref + identity behavior — without the validation schema

### before / after

| | before | after |
|---|---|---|
| `z.toJSONSchema(Job.schema)` (raw) vs `z.toJSONSchema(Job.contract)` (stamped) | anonymous `{ type: object, properties }` | same shape **+** `x-domain-object` pragma |
| consumer codegen | nameless types, can't de-dupe, no keys | named `SvcJobsJob`, de-duped, keys intact |
| identity across the wire | lost at the serialization boundary | survives serialize → json-schema → parse |

### the "aha"

the dobj's identity already lives as statics on the class
(`primary`/`unique`/`alias`/`nested`). today they evaporate the moment a schema
crosses to json-schema. the aha: **stamp once at the schema boundary, and the
dobj's identity rides the wire for free** — exactly like `serialize` stamps `_dobj`
so a string can be re-hydrated into the right class.

## user experience

### usecases / goals

| persona | goal |
|---|---|
| upstream author | declare endpoints with `Dobj.schema` and have identity auto-ride to json-schema, zero extra boilerplate |
| downstream codegen (`sdk-aws-lambda`) | detect "this json-schema is a domain object named X", learn its keys, reconstruct a named dobj |
| library maintainer | reuse the `_dobj` mental model; one stamping concept across serialize + json-schema |

### contract inputs & outputs

- **input**: author-supplied `static schema` (zod only) + the statics
  (`primary`, `unique`, `alias`, `nested`, name).
- **output**: a zod schema that carries a `.meta({ 'x-domain-object': {...} })` pragma
  (zod-4 registry), such that `z.toJSONSchema()` emits the pragma as a custom keyword.
- **stamp set = requested set only**: `name`, `primary`, `unique`, `alias`, `nested`.
  the class also carries `updatable`, `readonly`, `metadata` statics — these are **not**
  stamped (not requested; the consumer does not re-validate or persist). resist the urge
  to stamp a static just because it exists.
- **non-goal**: consumers do **not** reconstruct the zod validation schema; the
  originator runtime-typechecks, consumers only re-carry identity + keys.

### what leveraging it looks like

```ts
// upstream — .contract carries the stamp; the author writes no extra code
const out = z.object({ job: Job.contract });
const json = z.toJSONSchema(out); // pragma present

// downstream — introspect
const pragma = json.properties.job['x-domain-object'];
// → { name: 'Job', primary: ['uuid'], unique: ['exid'], alias, nested }
```

### timeline

1. author writes the dobj once (statics already there).
2. at endpoint declaration, the stamped schema is read.
3. codegen converts to json-schema → pragma present.
4. consumer parses pragma → names + de-dupes + rebuilds keys.

## mental model

- **how a user describes it to a friend**: "the schema carries a little ID badge
  that says which domain object it is and what its keys are, so the other service
  can rebuild it by name."
- **analogy**: a shipping label (`x-domain-object`) stapled to the box (json-schema).
  the box contents (shape) are unchanged; the label says what's inside and how to
  file it.
- **precedent**: this is `_dobj` for json-schema. serialize stamps `_dobj: "Job"`
  into a string; this stamps `x-domain-object: {...}` into a schema.

| they say | we say |
|---|---|
| "ID badge / label / tag" | pragma / stamp / `x-domain-object` keyword |
| "the schema knows what it is" | identity marker carried via zod `.meta()` registry |
| "keys" | `primary` (surrogate) / `unique` (natural) |

## evaluation

### how well it solves the goals

- **names + de-dupes**: ✅ `name` in the pragma is the carrier the consumer needs.
- **keys**: ✅ `primary` / `unique` ride along for ref + identity reconstruction.
- **nested**: ✅ achievable if we stamp nested dobj **names** (not class refs).
- **follows `_dobj`**: ✅ same "stamp identity onto the serialized form" pattern.

### pros

- zero author boilerplate — identity rides for free.
- one consistent stamping concept (`_dobj` ↔ `x-domain-object`).
- non-mutating: zod-4 `.meta()` returns a fresh schema; author's schema untouched.
- consumer stays lightweight — no need to reproduce validation.

### cons / costs

- couples `.contract` to **zod 4** (`.meta()` + `z.toJSONSchema`). joi/yup authors cannot
  use `.contract` — it fails fast with a `ConstraintError` (they keep `.schema` for
  validation; they just can't cross the wire with identity).
- introduces a custom json-schema keyword (`x-domain-object`) — a small standard we own.
- a getter literally named `schema` was impossible (JS static-shadowing); resolved by the
  separate `.contract` name (see awkward §).

### `.contract` access contract — fail-fast

> to get a `.contract`, you must declare a **zod** `static schema`.

`.contract` *is* the schema + identity. no schema → no shape to identify → no contract.
since `.contract` is brand-new (nothing depends on it yet), it is strict from day one —
fail loud at access, never silently downstream.

| access condition | behavior |
|---|---|
| no `static schema` declared | **`ConstraintError`** (caller-fix, exit 2): "`.contract` requires a `static schema`" |
| schema is **joi/yup**, not zod | **`ConstraintError`** (caller-fix, exit 2): "contract requires a **zod** schema; joi/yup cannot carry json-schema identity" |
| schema is zod | returns the stamped schema (the contract) |

both failure modes are caller-must-fix and surface a helpful hint at the access site —
no failhide, no surprise pragma-absent bug downstream (`rule.forbid.surprises`).

### edgecases & pit-of-success

| edgecase | pit-of-success contract |
|---|---|
| dobj has no keys (plain `DomainObject` w/ schema) | stamp `name` (+ `alias` if set) only; omit absent key fields |
| `nested` holds class refs | stamp nested **names**, never the constructor objects |
| author mutates returned schema | `.meta()` returns a fresh schema; cache per-class to stay idempotent |
| consumer on zod 3 | pragma is plain json — readable without zod at all |

## open questions & assumptions

### assumptions

1. consumers want **names + keys**, not the validation schema (stated in wish).
2. zod 4 is acceptable as the stamp engine (`.meta()` + `z.toJSONSchema`).
3. `x-domain-object` is an acceptable keyword name (wish invites collaboration on shape).
4. **class `name` is a stable identity carrier.** the pragma `name` = `constructor.name`,
   the same string `serialize` already trusts for `_dobj`. under minification / bundlers,
   `class Job` can mangle to `class t` — a real risk for the canonical type identity.
   (`alias` does **not** rescue this — see below — it's a different field; the mitigation
   is to keep dobj class names un-mangled, as `serialize`/`_dobj` already assume.)
5. **downstream tools preserve unknown `x-` keywords.** the pragma only survives if every
   json-schema tool between originator and consumer passes `x-domain-object` through. the
   `x-` vendor-extension convention favors this, but strict normalizers may strip it —
   validate with the `sdk-aws-lambda` side, who own the pipeline.

### open questions (triaged)

each tagged `[answered]` (decided now), `[research]` (answer in research/execution),
or `[wisher]` (needs wisher/consumer input).

1. **`[answered]` can a static getter literally named `schema` work?** no — a subclass
   `static schema = z.object()` data property shadows any inherited base-class accessor.
   answered by logic now; the design must use a non-clobbering shape. (the *choice* of
   shape is q2.)
2. **`[answered]` which non-clobber shape for the public api? → `Job.contract`.**
   wisher decided: a static getter named **`.contract`** on the base class that returns
   the stamped schema. used inline where a zod schema is expected:
   ```ts
   z.object({ surfboard: SeaturtleSurfboard.contract })
   ```
   - **why `.contract`**: a domain object *is* a contract — it declares "i am `Job`, my
     identity is `primary`/`unique`, my shape is this schema." that bundle (identity +
     keys + schema) is exactly what crosses the wire. the name's promise = the payload.
   - **why not `.schema`**: a subclass `static schema = z.object()` data property shadows
     any inherited base-class accessor (JS class-field `[[Define]]` semantics) — the
     getter would never fire.
   - **why not `.shape`**: zod already defines `.shape` as the *field map only* (no
     identity, no keys) — to name this `.shape` would understate the payload.
   - **why a getter, not a free function**: `.contract` reads as the dobj's own declared
     promise at the call site; the `serialize(x)` free-function precedent was the
     fallback if a non-clobber name could not be found — it can.
   - **ubiqlang note**: `contract` also names the `src/contract/` layer and procedure
     I/O, but those are *contracts of operations*; this is the *contract of a domain
     object* — the same idea (a declared promise) applied to a different subject. the
     receiver (`Job.`) disambiguates. coherent extension, not a clash by overload.
3. **`[answered]` keyword name → `x-domain-object`.** wisher decided.
   - **why `x-` prefix is mandatory (not style)**: json-schema is an open standard other
     tools read. the `x-` prefix is the agreed **vendor-extension** space — tools
     ignore-but-preserve `x-` keywords rather than reject/strip unknowns. a bare key like
     `_dobj` has no such protection (a strict validator could flag/strip it). so the
     pragma keeps the `x-` prefix to stay in the safe space (ties to assumption #5).
   - **why not reuse `_dobj`**: `_dobj` is *our* serialize-string format, which we fully
     own (`serialize.ts:113`, `deserialize.ts:94`); `_`-prefix is our metadata convention
     there. `x-domain-object` is the **same idea** (stamp dobj identity onto the
     serialized form) expressed in **json-schema's** dialect. same concept, two
     host-format conventions — they cannot share one spelling without one violating its
     host's rules.
   - **constraint**: the keyword **must keep the `x-` prefix**. the suffix spelling
     (`x-domain-object`) is now fixed.
4. **`[answered]` nested representation → name-only.** wisher decided: the pragma
   represents nested dobjs by **name only**, e.g. `"nested": { "customer": "Customer" }`.
   the consumer learns *which* dobj is nested (enough to wire the reference by name); it
   does **not** receive the nested dobj's own keys via this field. `$ref`-style nested
   contracts are a possible later enhancement if a consumer needs nested keys inline —
   out of scope for now. ("ideally `.nested`" was soft in the wish; name-only is the
   simple, sufficient start.)
5. **`[answered]` zod dependency policy → bump baseline to zod 4.** wisher decided:
   upgrade the lib's zod baseline to zod 4 (not peer/opt-in). domain-objects currently
   declares `zod: ^3.23.8` as a **type-only devDependency** (`validate.ts:4` imports
   `import type` only; author supplies the schema), so the bump is lower-risk than a
   runtime-dep major. **mandatory ripple**: `isZodSchema` detects via the zod-3 internal
   `_refinement` (`validate.ts:18`), absent in zod 4 — detection must change to recognize
   zod-4 schemas, else validation silently no-ops for zod-4 authors.
6. **`[answered]` does `z.toJSONSchema()` emit unknown `.meta()` keys verbatim? → yes.**
   proven in the execution stone with a real `zod/v4` call: `z.object().meta({
   'x-domain-object': {...} })` then `z.toJSONSchema()` emits `x-domain-object` **verbatim
   at the top level** of the output. also confirmed `.meta()` returns a fresh schema (no
   mutation) and zod-4 schemas are detectable via the `.meta` function. research probe run
   green, then removed.
7. **`[answered]` `name` and `alias` are distinct fields — stamp both.** wisher
   clarified: these are **not** competing carriers of one identity. they serve different
   jobs and both ride in the contract:
   - **`name`** = canonical type identity (`constructor.name`) → names the reconstructed
     type, de-dupes across endpoints.
   - **`alias`** = colloquial reference (`'job'` or `{ singular, plural }`,
     `DomainObject.ts:80`) → devex in consumer-side contracts.
   the minification risk in assumption #4 applies to **`name`** only and is a separate
   concern (mitigate by un-mangled dobj class names); it is **not** a reason to prefer
   `alias` over `name`. both are always stamped.
8. **`[answered]` keyword preservation downstream → confirmed preserved.** wisher
   confirmed the `sdk-aws-lambda` pipeline preserves unknown `x-` keywords end-to-end
   (assumption #5 holds).

### already answered by groundwork (no longer open)

- **`.meta()` does not mutate** the source schema — verified (returns a new instance).

## groundwork

### external research

the wish references zod's `z.toJSONSchema()` and `.meta()` / registry.

- **verified**: installed zod is **3.25.76** (`node_modules/zod/package.json:3`). this
  version ships **both** apis: zod 3 at the default `.` export and zod 4 at the
  `./v4` subpath (`package.json` exports map `:46-95`: `.`, `./v3`, `./v4`, `./v4/core`, …).
- **verified**: `z.toJSONSchema()`, `.meta()`, and `z.globalRegistry` are **zod-4-only**
  and present in the installed source:
  - `toJSONSchema` → `node_modules/zod/v4/core/to-json-schema.d.ts` (re-exported via `v4/classic/external.d.ts`)
  - `.meta(data)` returns a new registered instance + `z.globalRegistry` →
    `node_modules/zod/v4/classic/schemas.d.ts:47-53`
- **verified**: the default `import { z } from 'zod'` in 3.25.x is still **zod 3** —
  the root `node_modules/zod/index.d.ts:1` re-exports `./v3/external.js`. so stamping
  requires the author's schema to be a **zod 4** schema (`zod/v4`).
- **constraint**: the repo declares `zod: ^3.23.8` (dev dep, `package.json:107`). the
  feature's zod-version policy is an open question (above).
- **not yet verified** (flagged for execution stone): whether `z.toJSONSchema()` emits
  unknown `.meta()` keys *verbatim* into output (the apis exist; the passthrough
  behavior must be proven with a real call, not assumed).

### internal research

the wish references the `_dobj` serialization pattern and the statics. verified:

- **`_dobj` precedent**: `serialize` stamps `stringifiableObj._dobj = obj.constructor.name`
  (`src/manipulation/serde/serialize.ts:113`); `deserialize` reads `obj._dobj` to look
  up the class from context (`src/manipulation/serde/deserialize.ts:94-110`). this is the
  exact pattern to mirror for json-schema.
- **`.schema` static**: `public static schema?: SchemaOptions<any>`
  (`src/instantiation/DomainObject.ts:89`); consumed in the constructor for runtime
  validation (`:42-44`). it is **author-supplied data**, not currently a getter.
- **key statics to stamp**:
  - `alias` (`DomainObject.ts:80`), `nested` (`:126`), `metadata` (`:154`)
  - `primary` (`DomainEntity.ts:43`), `unique` (`:56`), `updatable` (`:65`), `readonly` (`:86`)
  - `primary` optional on `DomainLiteral` (`DomainLiteral.ts:41`)
- **identity marker precedent**: `MARK_AS_DOMAIN_OBJECT = Symbol.for('domain-objects/DomainObject')`
  carrying `VERSION` (`src/instantiation/markers.ts:7`, `DomainObject.ts:65`) — an extant
  "this is a domain object named/versioned X" tag we can echo.
- **zod detection gap (verified)**: `isZodSchema` detects via `schema._refinement`
  (`src/instantiation/validate/validate.ts:18`), a **zod-3 internal**. `_refinement`
  does **not** appear anywhere in `node_modules/zod/v4`, so this check will **not**
  recognize a zod-4 schema — detection must be revisited before stamping.
- **vocab to reuse**: `_dobj` (stamp key), `primary` / `unique` / `alias` / `nested`
  (key names), `getUniqueIdentifier` / `serialize` (free-function precedent for an
  accessor shape).

### internal — none beyond the above

no CLI stdout/error formats are involved; this is a library-surface feature.

## what is awkward?

1. **"a `.schema` getter" fights JS static-shadowing.** the wish's literal phrasing
   ("add a `.schema` getter ... that returns the author-supplied `static schema`
   stamped") can't be a static getter *named* `schema`, because the author's
   `static schema = z.object()` data property shadows it on the subclass. the design
   must pick a non-clobbering shape (free function or differently-named accessor).
   this is the single biggest design fork.

2. **zod-version coupling.** the feature only works when the author's schema is zod 4,
   yet the lib's baseline is zod 3. joi/yup are silently unsupported. that's a real
   capability cliff — graceful no-op for non-zod-4 keeps users in a pit of success but
   may surprise someone expecting a stamp.

3. **a custom json-schema keyword is a tiny private standard.** `x-domain-object` only
   means something to consumers who know to look for it. that's fine for the
   `sdk-aws-lambda` ↔ domain-objects pairing, but it's a coupling both sides must agree on.

4. **`nested` is class references, not data.** stamping must translate nested
   constructors → names; a naive stamp would try to serialize class objects. the
   "stamp names, not classes" rule is easy to get wrong.
