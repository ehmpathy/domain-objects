# blocker.1: Missing type narrowing for nested readonly in HasReadonly

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.require.behavior-intent-coverage.md

**locations**:
- src/manipulation/HasReadonly.type.ts:80-95
- src/manipulation/HasReadonly.type.test.ts:120-145

The vision requires 'narrow type after a fetch' where nested readonly fields become required inside hasReadonly guards (e.g. fetched.network.interface.privateIp typed as required, not undefined). The implementation explicitly defers this: HasReadonly only narrows flat metadata/readonly keys and ignores dot-paths (they are not keyof the instance). This leaves a core UX requirement from the vision unaddressed.

**snippet**:
```ts
  // nested (dot-path) readonly keys (e.g. `network.interface.privateIp`) are verified at runtime by `hasReadonly`,
  // but are NOT narrowed at the type level here. flat keys (metadata + top-level readonly) are narrowed as usual;
  // nested dot-path keys are gracefully ignored by this type (they are not a `keyof` the instance, so the
  // `keyof TInstance & ...` intersection filters them out). a type-level nested narrow was attempted but deferred:
  // the recursive mapped type it requires (`Omit<T, head> & { rewrite }`) is not provably assignable back to the
  // instance type, which breaks the `hasReadonly` type-guard's `obj is HasReadonly<TDobj>` contract.
```

# blocker.2: Incomplete fail-fast for invalid nested readonly paths

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.require.behavior-intent-coverage.md

**locations**:
- src/manipulation/omitReadonly.ts:55-70
- src/manipulation/hasReadonly.ts:25-40
- src/manipulation/omitReadonly.test.ts:280-310

Vision requires fail-fast at omit time (rather than no-op) when a readonly path 'doesn't trace through `static nested`'. Current logic only throws when a continuing path hits a present non-domain value. If a path segment is absent (e.g. 'network.foo.bar' where 'foo' not present), omit silently no-ops (no throw, object unchanged) and hasReadonly returns false with no error. This violates the 'fail-fast rather than no-op' requirement and leaves untested code paths for invalid paths.

**snippet**:
```ts
  // terminal value with no further nested paths => no nested keys to omit, return it
  if (nestedPaths.length === 0) return thisValue;

  // terminal value that is absent (declared-but-unresolved) => no value to omit yet, return it
  if (thisValue === null || thisValue === undefined) return thisValue;

  // terminal value that is present but not a domain object => misconfiguration, fail-fast
  // (a nested readonly path must traverse declared `static nested` domain objects so the path can be followed)
  throw new UnexpectedCodePathError(
    'omitReadonly: a nested readonly path points through a value that is not a domain object. ensure the sub-object is declared in `static nested` so it hydrates into a domain object instance.',
    { thisValue, nestedPaths },
  );
```
