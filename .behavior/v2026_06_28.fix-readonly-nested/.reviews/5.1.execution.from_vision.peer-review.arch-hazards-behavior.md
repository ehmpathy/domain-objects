# blocker.1: Type vs runtime contract mismatch for default metadata keys

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/manipulation/HasReadonly.type.ts
- src/manipulation/hasReadonly.ts
- src/manipulation/HasReadonly.type.test.ts
- src/manipulation/hasReadonly.test.ts

The HasReadonly type definition falls back to HasMetadata defaults (requiring id/uuid/createdAt/updatedAt etc.) when no explicit `public static metadata` is declared on the class. However, the hasReadonly runtime implementation always throws DomainObjectMetadataMustBeDefinedError if metadata is not explicitly declared on the class. This means code using HasReadonly<typeof SomeClassWithoutExplicitMetadata> type-checks and narrows correctly, but calling hasReadonly at runtime throws an error. This creates unpredictable behavior: passes static checks but fails at runtime, exactly the kind of hard-to-debug hazard described in the rule.

**snippet**:
```ts
    // fail fast if metadata is not explicitly declared
    const metadataKeys = (dobj as unknown as typeof DomainObject).metadata as
      | string[]
      | undefined;
    if (!metadataKeys)
      throw new DomainObjectMetadataMustBeDefinedError({
        domainObjectName: dobj.name,
        nameOfFunctionNeededFor: 'hasReadonly',
      });
```

# blocker.2: hasReadonly throws instead of returning false on invalid input

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- src/manipulation/hasReadonly.ts

The hasReadonly function is a type guard that should return boolean, but throws Error (or DomainObjectMetadataMustBeDefinedError) for non-DomainObject inputs or missing metadata declaration. Callers expect a boolean result from a 'hasX' check (as used in if (hasReadonly(...) (obj)) narrowing), but receive an exception instead. This is non-obvious control flow and hidden side-effect behavior that only manifests at runtime on certain inputs, violating predictable behavior under all conditions.

**snippet**:
```ts
    // ensure it's a domain object
    if (!isOfDomainObject(obj))
      throw new Error(
        'hasReadonly called on object that is not an instance of a DomainObject. Are you sure you instantiated the object?',
      );
```
