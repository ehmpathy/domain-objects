# blocker.2: type vs runtime mismatch for metadata and nested readonly keys

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/manipulation/HasReadonly.type.ts
- src/manipulation/hasReadonly.ts
- src/manipulation/hasReadonly.test.ts
- src/manipulation/HasReadonly.type.test.ts

HasReadonly type uses defaults from HasMetadata when no explicit metadata is declared on the class, and ignores nested dot-path readonly keys at the type level. However, the runtime hasReadonly always throws DomainObjectMetadataMustBeDefinedError if metadata is not explicitly declared and returns false for missing nested readonly (even if type allows the assignment). This creates unexpected runtime errors, lack of type narrowing for declared readonly, and friction hazards that are not clearly communicated or snapped. Users following type examples will hit runtime failures.

**snippet**:
```ts
type ExplicitMetadataKeysOf<TDobj> = TDobj extends { metadata: infer M }
  ? M extends readonly string[]
    ? string extends M[number]
      ? never
      : M[number]
    : never
  : never;

// later
if (!metadataKeys)
  throw new DomainObjectMetadataMustBeDefinedError({
    domainObjectName: dobj.name,
    nameOfFunctionNeededFor: 'hasReadonly',
  });

// type test example that would fail at runtime
const user: HasReadonly<typeof UserRecord> = {
  id: 1,
  uuid: 'abc-123',
  name: 'John',
  createdAt: new Date(),
  updatedAt: new Date(),
};
```

# blocker.3: error paths and blocked states not acceptance tested or snapped

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.friction-hazards.md

**locations**:
- src/manipulation/hasReadonly.test.ts
- src/manipulation/omitReadonly.test.ts

Multiple user-facing error paths (non-DomainObject input, missing metadata/readonly keys, nested path misconfigurations, assure failures) are either untested or tested only with toThrow(Class) or regex without snapshotting the actual error message or output. Blocked states like missing readonly are not clearly communicated via snapped outputs. Per the rule, every error path must be acceptance tested and snapped so reviewers can see the actual user experience; unsnapped friction will ship and frustrate users.

**snippet**:
```ts
expect(() =>
  hasReadonly({ of: DeclaredAwsRdsCluster }).assure(cluster),
).toThrow(AssureIsOfTypeRejectionError);

// similar patterns
expect(() => hasReadonly({ of: Coordinate })(coord)).toThrow(
  DomainObjectMetadataMustBeDefinedError,
);
expect(() => omitReadonly(instance)).toThrow(
  /nested readonly path points through a value that is not a domain object/,
);
```
