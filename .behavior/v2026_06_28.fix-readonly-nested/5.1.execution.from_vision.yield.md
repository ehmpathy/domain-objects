## decisions / outcomes

- **getReadonlyKeys**: no signature change — returns the raw `readonly` array (now may contain dot-paths); the split happens inside omitReadonly.
- **serialize**: no change — drift correctness holds because omit executes first (vision assumption #4, verified). drift-equality test added.
- **fail-fast**: two guards. the framework's `assertDomainObjectIsSafeToManipulate` catches a path through a plain (non-`nested`) object with a helpful message; a custom guard in omitReadonly catches a path through a primitive.
- **HasReadonly type-narrow — DEFERRED** (per wisher "runtime fine unless easy; try it though"):
  - attempted a recursive mapped type (`Omit<T, head> & { rewritten nested }`).
  - it failed type-check: the result is not provably assignable back to `InstanceType<TDobj>`, which breaks `hasReadonly`'s type-guard signature (`obj is HasReadonly<TDobj>`).
  - reverted to the flat-only `HasReadonly`. nested dot-path keys are gracefully ignored by the type (not a `keyof`, so filtered by `keyof TInstance & ...`) — flat keys still narrow, nested keys are runtime-verified by `hasReadonly`.
  - documented in the type's jsdoc + 2 type tests prove the graceful baseline (flat narrow holds, nested keys don't break the type).

## peer-review fixes (round 2)

peer review surfaced 12 blockers. addressed:
- **immutability** — `splitReadonlyPaths` rewritten from `.push()`/assignment to an immutable `reduce`.
- **fail-fast (deeper)** — now fails fast when a nested path head is absent from the object (typo'd path) OR descends into an absent/primitive value; `null` stays a legitimate absent subtree. added typo-path + primitive tests.
- **failloud** — prior plain `new Error` in `hasReadonly` upgraded to `UnexpectedCodePathError` with `{ obj }` context.
- **HasReadonly nested narrow — NOW IMPLEMENTED** — earlier `Omit<T,head> & {…}` broke the type-guard subtype contract; switched to an intersection overlay `T & DeepRequiredAtPaths<T,paths>` (an intersection is provably a subtype of `T`), so nested readonly fields narrow to required AND the type-guard contract holds. type tests rewritten to assert the narrow (positive + 2 `@ts-expect-error`).
- **deep coverage** — added a 3-level nested omit test.

### remaining (prior `hasReadonly` design — flagged for wisher, not unilaterally changed)
- `DomainObjectMetadataMustBeDefinedError`: reviewer wants a "standard" class, but it already extends `HelpfulError` with full context + actionable example — a proper domain HelpfulError. defensible as-is.
- `hasReadonly` throws (vs returns false) on non-domain-object / undeclared-metadata: deliberate prior type-guard design; changing it alters public behavior + breaks tests.
- type-vs-runtime metadata mismatch (type uses HasMetadata defaults; runtime requires explicit metadata): prior inconsistency, not introduced here.
