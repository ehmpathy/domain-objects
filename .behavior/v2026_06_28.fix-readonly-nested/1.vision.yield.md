# 1. vision — nested readonly attributes

> wish: support readonly attributes that live inside nested domain literals, declared from the entity grain via the attribute access chain (e.g. `network.interface.privateIp`).

---

## the outcome world

### day-in-the-life

a domain modeler at declastruct-aws models an ec2 instance. its network is a cohesive concept made of nested literals:

```ts
network: {
  subnet,
  security: { groups },
  interface: {
    publicIpEnabled,    // settable by the user
    sourceDestChecked,  // settable by the user
    privateIp,          // resolved by aws (persistence-set)
    publicIp,           // resolved by aws (persistence-set)
  },
}
```

they want `privateIp` and `publicIp` to live where they belong — under `network.interface` — yet still be treated as readonly (omitted from drift comparison, required after fetch).

today they can't. so they hoist `privateIp` to the entity top level, splitting one concept across two tiers purely to satisfy a framework limit.

### before / after

**before** — readonly only declarable flat on the entity:

```ts
class DeclaredAwsEc2Instance extends DomainEntity<...> {
  public static readonly = ['privateIp'] as const; // hoisted out of network.interface 😞
  public static nested = { network: AwsNetwork };
}
// privateIp lives at the wrong tier; network.interface is incomplete
```

**after** — readonly declarable along the access chain, from the entity grain:

```ts
class DeclaredAwsEc2Instance extends DomainEntity<...> {
  public static readonly = [
    'network.interface.privateIp',
    'network.interface.publicIp',
  ] as const; // 🎉 lives in its natural sub-scope
  public static nested = { network: AwsNetwork };
}
// network.interface now holds all its attributes; the entity owns which are persistence-set
```

### the "aha" moment

> readonly is a **fact the entity knows about its own persistence**, not a fact the literal knows about itself.

a `Address` / `AwsNetworkInterface` literal is stateless and reusable across many domains — in one domain `privateIp` might be user-supplied, in another it's aws-resolved. so the literal must stay silent. the **entity** declares the readonly truth and we **follow the access chain** down into the nested literal to apply it.

## user experience

### usecases & goals

| usecase | goal |
|---------|------|
| model a resource with nested persistence-set fields | keep the domain model cohesive (no forced hoisting) |
| plan-vs-apply drift detection (declastruct) | nested readonly fields excluded from the diff, so declared-vs-resolved doesn't falsely register drift |
| narrow type after a fetch | nested readonly fields become required, like flat readonly today |

### contract inputs & outputs

**input** — entity declares nested readonly via dotted access-chain keys:

```ts
public static readonly = ['host', 'network.interface.privateIp'] as const;
//                          ^flat (works today)  ^nested (new)
```

**outputs** — the readonly functions follow the chain:

```ts
// omitReadonly: recurses the chain and drops the nested readonly field
const omitted = omitReadonly(instance);
omitted.network.interface.privateIp; // undefined ✅ (dropped)
omitted.network.interface.publicIpEnabled; // preserved ✅ (settable)

// hasReadonly: checks nested readonly values are defined
if (hasReadonly({ of: DeclaredAwsEc2Instance })(fetched)) {
  fetched.network.interface.privateIp; // typed as required (not undefined) ✅
}

// serialize / drift: unchanged surface, correct behavior
serialize(omitReadonly(declared)) === serialize(omitReadonly(resolved)); // true ✅ no false drift
```

### timeline

1. modeler adds dotted keys to `static readonly`
2. `omitReadonly` / `hasReadonly` / `HasReadonly` honor them automatically
3. declastruct picks it up on a version bump — **no declastruct code change** beyond the bump

## mental model

### how a user describes it to a friend

> "readonly walks down into your nested objects now. you point at `network.interface.privateIp` from the entity and the framework follows that path to omit it during drift detection and require it after a fetch."

### analogies / metaphors

- **access chain** — like an object access path (`a.b.c`); the readonly declaration *is* the path you'd type to reach the field.
- **the entity is the source of truth** — the literal is a dumb, reusable shape; the entity is what knows its lifecycle. readonly flows *from* the grain that has state *to* the grains that don't.

### vocab — theirs vs ours

| they say | we say |
|----------|--------|
| "nested readonly field" | nested readonly attribute |
| "the path to the field" | attribute access chain |
| "stuff aws fills in" | persistence-set / readonly (non-metadata) |
| "diff" | drift detection |

## evaluation

### how well it solves the goal

- **directly** solves the cohesion problem: nested persistence-set fields live in their natural sub-scope.
- **keeps literals stateless**: declaration stays on the entity; literal classes untouched and reusable.
- **backwards compatible**: flat keys (`'host'`) keep working; dotted keys are purely additive.
- **downstream-free**: declastruct benefits via omitReadonly/serialize with only a version bump.

### pros

- one cohesive domain concept stays in one place
- no new top-level API — extends the `static readonly` array we already have
- the dotted notation matches the issue's own language (`network.interface.privateIp`)

### cons / costs

- `static readonly` becomes a mini path-language (must parse dots), not just keys
- `HasReadonly` type must become a recursive/path-aware mapped type — non-trivial typescript
- arrays-of-nested introduce a "which item?" question (path applies to *every* item)

### edgecases & pit-of-success

| edgecase | proposed handling |
|----------|-------------------|
| flat readonly key (`'host'`) | unchanged — no dot, applied at top level |
| nested literal in an array (`network.interfaces.privateIp`) | apply path to every array item (recursion already maps arrays) |
| path points at a non-`nested` key | fail-fast at omit time: the entity must declare the sub-object in `static nested`, else we can't hydrate/recurse it |
| nested key undefined on a declared (not-yet-resolved) object | omit drops it on both sides → serialize equal → no false drift |
| deep chains (3+ levels) | recursion handles arbitrary depth by passing remaining path segments down |
| nested readonly field marked **required** in the literal's schema | must be **optional** in the literal's schema — omit reconstructs via `new Constructor(...)` which re-runs validation (`omitReadonly.ts:88` → `DomainObject.ts:42-44`). same implicit constraint flat readonly already has today (`omitReadonly.test.ts:169-201` keeps readonly schema fields `optional`). document this so modelers don't hit a reconstruct-time validation throw. |

## open questions & assumptions

### assumptions made

1. **dot-path syntax** is the chosen contract (`'network.interface.privateIp'`), as an additive extension of the `static readonly: readonly string[]` we already have. ✅ **confirmed by wisher (2026-06-28): "dotpath is best."** the structured-map alternative is dropped; we accept the dot-as-separator constraint (assumption #5) as a documented limit.
2. the nested sub-object is always declared in `static nested` (required for hydration already), so we can rely on `.nested` to know where to recurse.
3. only the **entity** grain declares nested readonly; literals/events never gain a `static readonly`. (matches the wish — literals are stateless/reusable.)
4. `serialize` itself needs **no change** — but for a subtle reason: serialize collapses each nested domain object to `getUniqueIdentifier` (`serialize.ts:118-126`), and for a literal that is "all non-metadata keys present on the instance" (`getUniqueIdentifier.ts:53-61`). because `omitReadonly` runs **first** and reconstructs the nested literal *without* the readonly key, serialize never sees the dropped field. correctness rides entirely on `omitReadonly` dropping nested readonly before serialize is called. (if anyone ever serializes with `lossless: true`, the nested object is kept whole — but the omit still removed the field, so it stays correct.)
5. **dot is a pure separator** — the dot-path syntax assumes no real attribute key contains a literal `.`. if one did, the parser would mis-split and silently target a nonexistent path → the field stays un-omitted → false drift, silently. low risk in the aws domain, but a real failure mode of dot-as-separator and a concrete point in favor of the structured-map alternative (no parse ambiguity).
6. **the path traverses declared `nested` domain objects** — `omitReadonly` only descends when `isOfDomainObject(value)` (`omitReadonly.ts:18`), which holds only when the entity declares the sub-object in `static nested`. a readonly path that points at a **plain** nested object (not a declared `nested` literal) would be silently ignored. therefore the build must **fail-fast** when a readonly path doesn't trace through `static nested`, rather than no-op.

### triaged questions

each tagged **[answered]** (closed now), **[research]** (verify in research/build phase), or **[wisher]** (needs wisher input).

- **[answered] syntax**: ✅ dot-path string keys — **wisher chose "dotpath is best" (2026-06-28).** structured-map dropped; dot-as-separator constraint (assumption #5) accepted as a documented limit.
- **[answered] array semantics**: ✅ apply to *every* element — **wisher confirmed (2026-06-28): "apply to every element obviously always."** dot-path has no index token and the current recursion already maps arrays uniformly (`omitReadonly.ts:21-22`). no per-index target; no `[]`-wildcard syntax needed.
- **[answered] `HasReadonly` scope**: ✅ **wisher confirmed (2026-06-28): "only runtime support for now is fine, unless its easy. try it though."** → ship the **runtime** `hasReadonly` (nested path-walk) for sure; **attempt** the **type-level** nested-required narrow (`HasReadonly`) and keep it **only if it proves cheap**, else defer to a follow-up. type-level narrow is a recursive mapped type over dotted paths and is not needed for the drift correctness that motivates the issue.

### research needed externally

- **[research]** confirm declastruct's drift path consumes `omitReadonly` + `serialize` exactly as the issue states — and whether it serializes with `lossless` true/false (claimed, not yet verified against declastruct source). settle before we rely on "version-bump-only."

- `src/manipulation/omitReadonly.ts:16-26,70-88` — already recurses into nested domain objects + arrays, but nested recursion (`recursivelyOmitReadonlyFromObjectValue`) only strips the nested object's **own** metadata. **must thread the entity's nested readonly paths down into the recursion.**
- `src/manipulation/omitReadonly.test.ts:241-322` — the `nested` describe block proves metadata is stripped from nested literals; gives the test shape to extend for nested *readonly*.
- `src/manipulation/hasReadonly.ts:67-77` — flat `every key defined` check. **must walk nested paths.**
- `src/manipulation/HasReadonly.type.ts:29-55,92-98` — `ReadonlyKeysOf` + `WithReadonly` operate on flat key unions. **must become path-aware to require nested fields.**
- `src/instantiation/DomainObject.ts:126` — `static nested` is the map we lean on to know where to recurse; `:94` confirms serialize/getUniqueIdentifier rely on static props.
- `src/manipulation/serde/serialize.ts:118-126` — **read & verified.** serialize collapses non-root nested entities/literals to `getUniqueIdentifier` when `lossless=false`. needs **no change** because omit runs first (see assumption #4); the collapse operates on the already-omitted instance.
- `src/manipulation/getUniqueIdentifier.ts:53-61` — **read & verified.** for a literal, returns all keys except metadata. confirms a nested literal can't self-exclude `privateIp` (it has no metadata/readonly path to it) — reinforcing why the entity-grain declaration is necessary.

## what is awkward?

- **`static readonly` overloads its meaning**: it goes from "a list of keys" to "a list of keys *and* access paths". a reader must now know dots are significant. mitigation: document clearly; the dot mirrors real property access so it reads naturally.
- **declaration/shape split**: the readonly *fact* lives on the entity while the *field* lives on the literal — correct by design (stateless literal), but a reader chasing `privateIp` finds it declared readonly far from where it's defined. this is inherent to the wish, not a flaw we can design away.
- **the typescript for `HasReadonly` with nested required paths is genuinely hard** — recursive mapped types over dotted string unions. there's a real risk the type-narrowing half is heavier than the runtime half; may warrant splitting type-narrowing into a follow-up if it balloons.
- **arrays + paths are ambiguous** without a wildcard token; `network.interfaces.privateIp` reads like a single field, not "every interface's privateIp". worth resolving the notation with the wisher early.
